# Test-Driven Development(TDD)

All feature development and bug fixes must follow the **TDD** workflow:

1. **Red** — Write a failing test that defines the expected behavior
2. **Green** — Write the minimum code to make the test pass
3. **Refactor** — Clean up the code while keeping tests green

## Rules

- Write tests **before** writing implementation code
- Each test should cover one specific behavior or scenario
- Run the test suite after every change to ensure no regressions
- Aim for meaningful coverage of success paths, edge cases, and error handling

## What NOT to Test

Do NOT write dedicated unit tests for classes that are pure boilerplate with no custom logic:

- **Exception subclasses** — only call `super(...)`, compiler validates constructors
- **POJOs / DTOs** — only contain fields, getters, setters, `equals`, `hashCode`, `toString` (especially those generated by Lombok `@Data` / `@Value`)
- **Enums without behavior** — only define constants
- **Delegation wrappers** — forward every call to another object unchanged

These classes are either compiler-checked, framework-generated, or already covered by JDK tests. Dedicated tests add maintenance cost with zero defect-catching value.

**DO** test these classes when they carry custom logic:

- POJO with validation in constructor or setter
- Exception with computed fields (`isRetryable()`, message formatting)
- Enum with behavior methods
- Wrapper that transforms or filters before delegating

When testing code that *uses* these classes, verify the contract via `assertThrows`, field assertions, etc. — that is meaningful coverage.
